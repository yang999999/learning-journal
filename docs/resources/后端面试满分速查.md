## 一、MySQL（必问，权重最高）

### Q1：MySQL 索引为什么用 B+Tree？为什么不用 B-Tree/Hash/红黑树？
**满分答法**：
- **B+Tree 优势**：①非叶子节点只存 key，单个节点容纳更多 key，树高低（3层存千万级数据），磁盘IO少；②叶子节点双向链表串联，范围查询只需顺序扫描；③所有查询落到叶子节点，路径长度稳定，性能可预测。
- **B-Tree 的问题**：每个节点都存 data，导致单节点 key 数少、树高更高；范围查询需要中序遍历回溯。
- **Hash 的问题**：只支持等值查询，不支持范围/排序/最左前缀；Hash 冲突多时性能退化。
- **红黑树的问题**：二叉树结构，树高随数据量线性增长（百万数据树高~20），磁盘IO太多。
- **追问**：三层 B+Tree 能存多少数据？假设 key 16字节，指针6字节，每页16KB，一个节点存约 16K/(16+6)≈727 个指针，三层 = 727×727×16 ≈ 850万行（按每行1KB算）。

### Q2：什么是聚簇索引和二级索引？回表是什么？覆盖索引是什么？
**满分答法**：
- **聚簇索引**：InnoDB 主键索引，叶子节点存**整行数据**。一张表只有一个（主键或隐式row_id）。
- **二级索引**：非主键索引，叶子节点存**主键值**（不是行数据）。
- **回表**：通过二级索引找到主键，再到聚簇索引查整行的过程，多一次 B+Tree 查询。
- **覆盖索引**：查询所需字段全部在二级索引里，不需要回表。例如 `SELECT name FROM user WHERE age=20`，若建索引 (age,name)，直接从二级索引返回 name，效率高。
- **索引下推（ICP）**：MySQL 5.6+，在存储引擎层用联合索引的字段先过滤，减少回表次数。
- **追问**：为什么建议用自增主键？①顺序插入，B+Tree 尾部追加，减少页分裂；②二级索引叶子节点只存主键值，主键越小二级索引越小。UUID 随机插入导致频繁页分裂+空间浪费。

### Q3：MVCC 是怎么实现的？RC 和 RR 有什么区别？
**满分答法**：
- MVCC = **Undo Log 版本链** + **ReadView** 实现非锁定读。
- **Undo Log版本链**：每次修改记录时，Undo Log 保留旧版本，用 trx_id 和 roll_pointer 串联成链表。
- **ReadView**：事务进行快照读时生成，包含 m_ids(活跃事务ID列表)、min_trx_id、max_trx_id、creator_trx_id。
- **可见性判断**：
  - trx_id == creator_trx_id → 可见（自己改的）
  - trx_id < min_trx_id → 可见（事务已提交）
  - trx_id >= max_trx_id → 不可见（ReadView 之后才开始）
  - 中间 → 在 m_ids 中则不可见（事务未提交），不在则可见（已提交）
- **RC vs RR**：RC 级别每次 SELECT 都生成新 ReadView（能读到其他事务最新提交）；RR 级别只在第一次 SELECT 生成 ReadView，后续复用（保证可重复读）。
- **追问**：RR 级别怎么解决幻读？快照读靠 MVCC 保证一致；当前读（SELECT FOR UPDATE/INSERT/UPDATE）靠 Next-Key Lock（行锁+间隙锁）锁住区间防止新数据插入。

### Q4：MySQL 的锁机制（行锁/表锁/间隙锁/Next-Key Lock）？死锁怎么排查？
**满分答法**：
- **表锁**：MyISAM 引擎，锁住整张表，并发差；InnoDB DDL时用。
- **行锁（Record Lock）**：锁住索引记录，走索引才加行锁，不走索引会退化为表锁。
- **间隙锁（Gap Lock）**：锁住索引之间的间隙（开区间），防止插入幻影记录，只在RR级别生效。
- **Next-Key Lock**：行锁+间隙锁组合（左开右闭区间），InnoDB 默认加锁方式。
- **死锁条件**：互斥+持有并等待+不可剥夺+循环等待。InnoDB 有死锁检测（wait-for graph），回滚代价小的事务。
- **排查**：`show engine innodb status` 看 LATEST DETECTED DEADLOCK；`information_schema.innodb_trx/innodb_locks` 看当前事务和锁。
- **预防**：①固定表访问顺序；②大事务拆小；③合理加索引避免锁范围扩大；④用等值查询代替范围查询减少间隙锁范围。

### Q5：Redo Log / Undo Log / Binlog 有什么区别？两阶段提交是怎么做的？
**满分答法**：
- **Redo Log（重做日志）**：InnoDB 引擎层，物理日志（记录"在哪个数据页改了什么"），WAL 机制，崩溃恢复用。循环写，固定大小。
- **Undo Log（回滚日志）**：InnoDB 引擎层，逻辑日志（记录反向SQL），回滚+MVCC用。
- **Binlog（归档日志）**：Server 层，逻辑日志（SQL语句或行变更），主从复制+数据归档用。追加写。
- **两阶段提交（UPDATE流程）**：
  1. 写 Undo Log，修改 Buffer Pool
  2. 写 Redo Log（prepare 状态）→ 刷盘
  3. 写 Binlog → 刷盘
  4. Redo Log 改为 commit 状态
- **为什么要两阶段提交？** 保证 Redo 和 Binlog 一致性。假设不用两阶段，Redo 写完后 Binlog 没写完就宕机，主库用 Redo 恢复有数据但从库 Binlog 没同步到，主从不一致。
- **追问**：刷盘策略？`innodb_flush_log_at_trx_commit=1`（每次提交刷redo，最安全）；`sync_binlog=1`（每次提交刷binlog）。双1配置保证零丢失但性能略差。

### Q6：一条 SQL 执行很慢，你怎么排查优化？
**满分答法（有体系）**：
1. **先看 EXPLAIN**：
   - `type`：system > const > eq_ref > ref > range > index > ALL，至少要 range 级别
   - `key`：实际用了哪个索引，NULL 说明没走索引
   - `rows`：预计扫描行数，越小越好
   - `Extra`：Using index（覆盖索引，好）、Using where（回表过滤）、Using filesort（额外排序，不好）、Using temporary（临时表，不好）
2. **看索引是否失效**：对索引列做函数/运算/隐式类型转换；LIKE '%xx'；违反最左前缀；OR含非索引列。
3. **看数据量**：单表数据量太大（>500万/1000万看具体行大小）考虑分库分表或分区。
4. **看是否锁等待**：`show processlist` 看 State；`innodb_trx` 看长事务。
5. **看 Buffer Pool 命中率**：`show global status like 'Innodb_buffer_pool_read%'`，命中率应>99%。
6. **慢查询日志**：`slow_query_log=1`，`long_query_time=1`，收集慢 SQL 统一优化。
- **追问**：深分页（`LIMIT 1000000,10`）怎么优化？①用覆盖索引+子查询 `WHERE id > (SELECT id FROM t LIMIT 1000000,1)`；②标签记录上一页最后ID；③ES 等搜索引擎。

## 二、Redis（必问，权重最高）

### Q1：Redis 为什么这么快？真的是单线程吗？
**满分答法**：
- **快的原因**：
  ①纯内存操作（微秒级）；
  ②IO多路复用（epoll）单线程处理大量连接，避免线程切换和锁开销；
  ③高效数据结构（SDS/quicklist/ziplist/skiplist/hashtable）；
  ④Redis 6.0 引入多线程IO（读写网络数据），但命令执行仍是单线程（保证线程安全）。
- **澄清**：不是完全单线程——后台有 bio 线程处理 close 文件、AOF fsync、key 过期删除等；Redis 6.0 后网络IO多线程，但命令执行还是单线程。
- **追问**：单线程怎么处理高并发？靠 epoll 事件循环+非阻塞IO，连接不阻塞，操作都在内存微秒级。

### Q2：缓存穿透、击穿、雪崩的区别和解决方案？
**满分答法**：
| 问题 | 现象 | 原因 | 方案 |
|------|------|------|------|
| **穿透** | 查询不存在的数据，打穿到DB | 缓存和DB都没有 | ①布隆过滤器；②缓存空值（短TTL）；③参数校验 |
| **击穿** | 热点key过期瞬间大量请求到DB | 热点key失效+高并发 | ①互斥锁（SETNX重建缓存，单线程回源）；②逻辑过期（永不过期，后台异步刷新） |
| **雪崩** | 大量key同时过期 或 Redis宕机 | TTL集中/单点故障 | ①TTL加随机偏移；②Redis高可用（Cluster/Sentinel）；③多级缓存；④熔断降级 |
- **追问**：布隆过滤器原理？位数组+多个Hash函数，插入时k次hash置位，查询时k次hash都为1才可能存在（有误判率，不会漏判）。不支持删除（Counting Bloom Filter 可解决）。

### Q3：Redis 分布式锁怎么实现才正确？Redlock 了解吗？
**满分答法**：
- **正确姿势**：
  - 加锁：`SET lock_key unique_value NX EX 30`（NX互斥、EX过期防死锁、unique_value标识持有者）
  - 解锁：Lua 脚本原子判断+删除（不能用DEL，防止误删别人的锁）
    ```lua
    if redis.call("get", KEYS[1]) == ARGV[1] then
        return redis.call("del", KEYS[1])
    else
        return 0
    end
    ```
  - **看门狗续期**：Redisson 的 watchdog 默认每 TTL/3（10s）自动续期到30s，业务宕机锁自动释放。
- **Redlock**：在N个独立Redis实例上同时加锁，多数派(N/2+1)成功才算加锁成功，解决单节点主从异步复制丢锁问题。
- **争议**：Martin Kleppmann 认为 Redlock 在时钟漂移/GC停顿下仍不安全；antirez 反驳。工业界实践：绝大多数场景单实例+看门狗够用；金融级强一致建议用 ZooKeeper/etcd（临时顺序节点+会话机制更可靠）。
- **追问**：Redis 分布式锁和 ZK 锁的区别？Redis 是 AP 思路（高性能，极端情况可能丢锁）；ZK 是 CP 思路（强一致，性能稍差），基于临时节点+watch。

### Q4：Redis 持久化机制 RDB 和 AOF 怎么选？
**满分答法**：
- **RDB**：定时快照，二进制压缩文件，恢复快；缺点是两次快照间数据可能丢失；`bgsave` fork子进程（COW写时复制）不阻塞主进程。
- **AOF**：追加日志，记录每个写命令，数据更安全（everysec策略最多丢1秒）；缺点文件大、恢复慢。重写（rewrite）压缩。
- **混合持久化（4.0+）**：AOF重写时前半段用RDB格式（快），后半段用增量AOF命令，兼顾恢复速度和数据安全。**生产推荐**。
- **追问**：bgsave 时 fork 会不会阻塞？fork本身会短暂阻塞（复制页表），但子进程通过COW共享内存，不阻塞主进程读写。内存越大fork越慢，要控制Redis单实例内存（建议<10GB）。

### Q5：Redis 内存淘汰策略有哪些？
**满分答法**：
- **不淘汰**：noeviction（默认，内存满了直接报错）
- **设置了过期时间的key**：volatile-lru（最近最少用）、volatile-lfu（最不经常用，4.0+）、volatile-ttl（快过期的）、volatile-random
- **所有key**：allkeys-lru（最常用）、allkeys-lfu（推荐，4.0+更精准）、allkeys-random
- **近似LRU实现**：Redis 不维护双向链表（省内存），而是随机采样N个key（默认5个）选最久未用的淘汰；LFU 也是近似，用 counter + 衰减时间。
- **追问**：LRU 和 LFU 的区别？LRU 看最近一次使用时间，容易被偶发访问污染；LFU 看访问频率，更能保留真正的热点key。

### Q6：Redis Cluster 原理？
**满分答法**：
- **哈希槽**：16384 个槽位，均匀分配给多台Master。key 通过 `CRC16(key) % 16384` 定位槽位。
- **节点通信**：Gossip 协议（PING/PONG），去中心化，节点间交换槽位/状态/故障信息。
- **故障转移**：Master挂了，其Slave通过Raft-like选举成为新Master。
- **限制**：不支持跨槽多key操作（MGET/MSET跨节点报错）；事务/Lua脚本中key必须在同一节点；可用 Hash Tag（`{order}:123`）让相关key落同一槽位。
- **扩容**：新增节点后通过 `redis-trib.rb reshard` 迁移槽位，迁移过程中key通过 ASK/MOVED 重定向。

## 三、消息队列（Kafka为主，高频）

### Q1：Kafka 为什么快/高吞吐？
**满分答法**：
- **顺序写磁盘**：Partition 日志是顺序追加写，省去磁头寻址，速度接近内存随机写。
- **零拷贝（sendfile）**：数据直接从磁盘通过 DMA 到网卡，不经过用户态，减少 CPU 拷贝和上下文切换。
- **页缓存（Page Cache）**：利用 OS 页缓存，写入先到 Page Cache 由 OS 异步刷盘；消费时热数据直接从内存读。
- **批量+压缩**：Producer 批量发送（batch.size/linger.ms），支持 Snappy/LZ4/ZSTD 压缩，减少网络IO。
- **分区并行**：Topic 分多个 Partition，Consumer Group 内多 Consumer 并行消费。
- **追问**：零拷贝是什么？传统 IO 路径：磁盘→内核缓冲→用户缓冲→Socket缓冲→网卡（4次拷贝4次上下文切换）；sendfile 路径：磁盘→内核缓冲→网卡（DMA gather copy，2次拷贝2次切换）。

### Q2：Kafka 怎么保证消息不丢失？
**满分答法**：
| 环节 | 可能问题 | 解决方案 |
|------|----------|----------|
| Producer | 网络抖动发送失败 | acks=all（等待所有ISR确认）+ retries重试；或本地消息表+补偿 |
| Broker | Leader写了但未同步就宕机 | acks=all + `min.insync.replicas≥2`（至少2副本确认） |
| Consumer | 没处理完就提交offset | 手动提交 offset（enable.auto.commit=false），业务处理完再commit |
- **ISR机制**：与Leader保持同步的Follower列表（`replica.lag.time.max.ms`内有心跳/消息同步），Leader从ISR选举，保证不丢。
- **追问**：acks=0/1/all 区别？0不等待确认（最快，可能丢）；1等Leader写完（默认，Leader挂了丢数据）；all等所有ISR写完（最安全）。

### Q3：Kafka 怎么保证消息顺序？
**满分答法**：
- **全局有序**：Topic 只设1个Partition，单Consumer单线程消费——性能极差，不推荐。
- **局部有序（推荐）**：业务维度相同key的消息发同一个Partition（如按orderId路由），一个Partition同一时间只被一个Consumer消费。
- **消费端注意**：Consumer内如果用多线程异步处理会破坏顺序，需要内存中按key分发到不同队列，每个队列串行处理。
- **追问**：如果有重平衡（Rebalance）怎么办？StickyAssignor 减少重平衡时分区移动；静态成员（`group.instance.id`）减少不必要的重平衡。

### Q4：Kafka 怎么处理重复消费？幂等怎么设计？
**满分答法**：
- **为什么重复**：网络抖动导致Producer重试（broker已存但ack丢了）；Consumer处理完还没commit offset就挂了。Kafka 默认 at-least-once 语义。
- **幂等方案（在业务端做）**：
  ① **数据库唯一键**：插入用唯一约束，重复直接报错（最简单）。
  ② **去重表**：建去重表（msg_id唯一键），先插去重表成功再处理业务（同一事务）。
  ③ **状态机幂等**：订单状态只能单向流转（待支付→已支付→已发货），重复消息无法改变已完成状态。
  ④ **Redis setnx**：分布式场景用msgId做幂等。
- **Kafka幂等Producer**：`enable.idempotence=true`（默认开），Producer给每条消息加PID+sequence number，Broker端去重，只保证单分区单会话不重复。
- **追问**：Kafka事务怎么实现跨分区原子写？通过事务协调器（Transaction Coordinator）+ 两阶段提交，保证 Consumer 端 read_committed 时要么读到全要么读不到，配合 Exactly-Once 语义。

### Q5：消息积压怎么处理？
**满分答法**：
1. **先定位原因**：是消费变慢（GC/DB慢/下游依赖故障）/消费失败循环重试/上游突发流量。
2. **临时扩容**：增加Consumer数量（不能超过Partition数，多了空转）；还不够则考虑扩容Partition。
3. **紧急处理**：如果积压太大影响线上，临时建一个新topic（更多Partition），把积压消息转发到新topic用更多消费者并行消费，处理完恢复。
4. **长期方案**：监控 consumer lag 告警；消息处理失败入死信队列避免阻塞；异步处理解耦；消费者逻辑优化。
- **追问**：怎么监控消息积压？Kafka 自带 JMX 指标 `RecordsLagMax`；Burrow 工具；或用 kafka-consumer-groups.sh 查看 LAG。

## 四、分布式系统（高频）

### Q1：CAP 定理？为什么分布式不能同时满足 CA？
**满分答法**：
- **C（Consistency）**：所有节点数据一致；**A（Availability）**：每个请求都能收到非错误响应；**P（Partition Tolerance）**：网络分区时系统仍可工作。
- **分布式下 P 不可避免**（网络一定会故障），只能在 C 和 A 之间选：
  - **CP**（保一致性牺牲可用性）：网络分区时少数节点拒绝服务——ZooKeeper/etcd/HBase/HDFS。
  - **AP**（保可用性牺牲强一致）：每个节点都响应，但可能返回旧数据，最终一致——Eureka/Cassandra/DNS/大多数电商业务。
- **误区**：CA 系统在分布式下不存在（单机才有CA）。
- **追问**：BASE 理论？Basically Available（基本可用）、Soft state（软状态）、Eventually consistent（最终一致性）。AP的工程落地指导思想。

### Q2：Raft 协议怎么工作的？
**满分答法**：
- **三种角色**：Follower、Candidate、Leader；Leader 处理所有写请求。
- **Leader 选举**：
  - Follower Election Timeout超时后变Candidate，term+1，给自己投票并向其他节点发RequestVote。
  - 获得多数票成为Leader；心跳维持权威。
  - 选举限制：只有拥有最新最全日志的候选者才能当选（比较最后一条日志的term和index）。
- **日志复制**：
  - Client写请求→Leader追加日志→通过AppendEntries RPC复制给Follower→多数确认后Leader commit并返回Client→下一次心跳通知Follower commit。
- **安全性保证**：Leader-Completeness（已提交日志一定在未来Leader中）；只从Leader到Follower单向同步日志。
- **追问**：Raft 和 Paxos 的区别？Paxos难理解难实现，Multi-Paxos没有标准实现；Raft为可理解性设计，拆解为选举/复制/安全三个子问题，工业界广泛使用（etcd/TiKV/Consul）。

### Q3：分布式 ID 方案？Snowflake 原理和问题？
**满分答法**：
- **方案对比**：
  - **UUID**：本地生成，无网络开销；但无序，做主键B+Tree插入导致页分裂，不推荐。
  - **数据库自增**：简单；但单点/性能瓶颈。
  - **数据库号段模式**（Leaf-segment）：从DB批量取ID段（如每次拿1000个），内存分配。性能好，分布式友好。
  - **Redis INCR**：高性能；但Redis持久化有丢ID风险。
  - **Snowflake雪花算法**：最常用——64bit = 1符号位 + 41时间戳(ms) + 10机器ID + 12序列号(每ms每节点4096个)。
- **Snowflake问题**：
  ① 时钟回拨（NTP校时可能回拨）：等待/拒绝/使用扩展位记录回拨次数。
  ② 机器ID分配：用ZK/DB/手动配置保证唯一。
  ③ 41位时间戳用到2039年（可自定义纪元延长）。
  ④ 序列溢出：同一ms超过4096个会等待下一ms。

### Q4：限流算法？令牌桶和漏桶的区别？
**满分答法**：
| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 固定窗口 | 单位时间计数 | 实现简单 | 边界突发（窗口交界处可能2倍流量） |
| 滑动窗口 | 细分子窗口滑动统计 | 平滑 | 内存占用高（Redis Zset实现） |
| 漏桶 | 请求入桶，固定速率流出 | 强平滑 | 应对突发流量差 |
| **令牌桶** | **固定速率放令牌，拿到令牌才通过** | **允许一定突发，生产最常用（Guava RateLimiter/Sentinel）** | 实现稍复杂 |
- **追问**：分布式限流怎么实现？Redis + Lua 脚本（原子操作：取当前时间→清过期桶→判断令牌数→扣减，单机版扩到分布式）；或 Sentinel/Resilience4j 网关层限流。

### Q5：服务熔断和降级？Sentinel 原理？
**满分答法**：
- **熔断**：下游服务故障比例达阈值时，快速失败（不调用下游），一段时间后半开探测，恢复则关闭。状态机：Closed → Open → Half-Open。
- **降级**：整体负载高时，关掉非核心服务（如推荐→改为热门榜单），保证核心功能可用。
- **隔离**：
  - 线程池隔离：每个服务独立线程池，一个服务挂不影响其他（Hystrix默认）。
  - 信号量隔离：计数器限制并发（轻量，不能异步超时）。
- **Sentinel vs Hystrix**：Hystrix 基于线程池隔离（开销大），已停更；Sentinel 基于滑动窗口+信号量，支持流控/熔断/热点参数/系统自适应保护，阿里开源国内主流。
- **追问**：怎么防止服务雪崩？①超时设置（不能无限等待）；②熔断降级；③限流；④舱壁模式（隔离）；⑤异步解耦（MQ削峰）。

### Q6：分布式链路追踪原理？
**满分答法**：
- **核心概念**：TraceID（全局唯一请求ID）、SpanID（当前调用单元）、ParentSpanID（父调用），形成树形调用链。
- **传递**：HTTP Header / gRPC Metadata / MQ 消息属性中透传。跨线程/跨进程/跨MQ都要做好Context传递。
- **OpenTelemetry**：CNCF 标准，统一 Trace/Metrics/Logs 的 API 和 SDK，取代了 OpenTracing+OpenCensus。
- **工具**：Jaeger/Zipkin 展示；SkyWalking（Java Agent 字节码注入，无侵入）。
- **追问**：为什么 MQ 场景链路追踪难？消息是异步的，Producer 发送后就返回了，Consumer 消费时上下文已经断了；需要在消息头传递 TraceContext，Consumer 取出后作为父 Span 创建新的 Span。

## 五、分布式事务

### Q1：分布式事务方案对比？你们项目怎么选的？
**满分答法**：
| 方案 | 一致性 | 性能 | 侵入性 | 适用场景 |
|------|--------|------|--------|----------|
| 2PC/XA | 强一致 | 差（同步阻塞） | 低（中间件代理） | 传统单体跨库、低并发 |
| TCC | 最终一致 | 好 | 高（写Try/Confirm/Cancel） | 资金等核心短流程 |
| Saga | 最终一致 | 好 | 中（写补偿） | 长流程（旅行/订单流转） |
| **本地消息表+MQ** | **最终一致** | **好** | **中** | **非核心链路最常用（通知/积分/短信）** |
| **RocketMQ事务消息** | 最终一致 | 好 | 低（MQ保证） | 电商下单等半消息场景 |
| Seata AT | 最终一致 | 中 | 低（自动生成反向SQL） | 快速落地，对性能不极端敏感 |

- **选方案原则**：能不做分布式事务就不做（设计层面避免）；必须做时优先最终一致（异步+补偿）；只有资金类核心场景才考虑TCC/强一致。
- **追问**：TCC 的空回滚、悬挂、幂等怎么处理？事务控制表（xid+branch_id+状态）：
  - 空回滚：Cancel时无TRY记录，直接记录空回滚标记返回。
  - 悬挂：Try执行时检查是否已Cancel，已Cancel则拒绝执行。
  - 幂等：Confirm/Cancel先查状态，已完成直接返回。

### Q2：Seata AT 模式原理？
**满分答法**：
1. **一阶段**：拦截业务SQL，查询前镜像(before image)→执行业务SQL→查询后镜像(after image)→把前后镜像和SQL写入UNDO_LOG表（本地事务内提交）→申请并获取全局锁→提交本地事务→释放本地锁。
2. **二阶段提交**：异步删除 UNDO_LOG（快速）。
3. **二阶段回滚**：根据 UNDO_LOG 生成反向SQL（after image→before image），先校验脏写（对比当前数据和after image），执行回滚。
- **注意**：AT模式默认读未提交隔离级别，需要`SELECT FOR UPDATE`获取全局锁才是读已提交。
- **全局锁**：TC（事务协调器）管理，防止多个全局事务同时修改同一行。
- **追问**：AT 和 TCC 模式的区别？AT自动生成反向SQL，无侵入但只能在关系型数据库且有SQL解析限制；TCC侵入性大但适用更广（可跨数据库/跨服务/非关系型）。

## 六、Docker/K8s（常问基础）

### Q1：Docker 容器和虚拟机的区别？
**满分答法**：
| 维度 | 容器 | VM |
|------|------|-----|
| 隔离 | 进程级（共享宿主内核，Namespace+Cgroups） | OS级（独立内核，Hypervisor） |
| 启动 | 毫秒~秒 | 秒~分钟 |
| 镜像 | MB级 | GB级 |
| 性能 | 接近原生 | 10-20%开销 |
| 安全 | 弱（共享内核，逃逸风险） | 强 |

- **Namespace（6种）**：PID（进程）、NET（网络）、MNT（挂载点）、UTS（主机名）、IPC（进程通信）、USER（用户）→做**隔离**。
- **Cgroups**：限制CPU/内存/IO/PID数量 →做**资源限制**。
- **UnionFS（OverlayFS）**：分层镜像（只读层+读写层）→做**镜像存储**。
- **追问**：容器内PID 1是什么？容器的init进程，挂了容器就退出。PID 1 有特殊职责（回收孤儿），且Linux信号默认不处理（需要业务捕获SIGTERM）。

### Q2：怎么优化 Docker 镜像大小？
**满分答法**：
1. **选小基础镜像**：alpine（5MB）/distroless（谷歌空镜像）/scratch。
2. **多阶段构建**：builder阶段装编译工具打包，runtime阶段只COPY最终产物。
3. **合并RUN指令**：`&&`连接，每层文件变更产生新层，安装完立刻清理缓存（apt-get clean && rm -rf /var/lib/apt/lists/*）。
4. **用COPY不用ADD**：ADD会自动解压有安全隐患；COPY更明确。
5. **.dockerignore**：排除 node_modules/.git 等无关文件。
6. **dive 工具**：分析每层内容找冗余。

### Q3：K8s 中 Pod 是什么？为什么有 Pod 这层？
**满分答法**：
- Pod是K8s最小调度单位，包含1~N个容器，总是一起调度/生死/部署在同一节点。
- **为什么需要Pod**：
  ①有些应用天然要紧耦合（web+日志采集sidecar/服务网格代理Envoy）；
  ②提供"逻辑主机"抽象，Pod内共享Network Namespace（同IP/localhost通信）和UTS/IPC；
  ③共享Volume存储。
- **Pause容器**：每个Pod有一个pause容器作为根容器，持有所有Namespace和Volume，业务容器加入进来。pause挂了整个Pod重建。
- **追问**：Deployment vs StatefulSet vs DaemonSet？Deployment无状态应用；StatefulSet有状态应用（稳定网络标识、稳定存储、有序部署）；DaemonSet每个节点跑一个（日志采集/监控Agent）。

## 📝 后端面试答题总原则

# M1 · MySQL 深度优化

## 一、核心知识点

1. **索引原理与设计**：B+Tree 结构、聚簇索引 vs 二级索引、联合索引最左前缀、覆盖索引、索引下推（ICP）、索引选择性
2. **事务与锁**：ACID、MVCC（Undo Log + ReadView）、行锁/表锁/间隙锁/Next-Key Lock、死锁检测
3. **日志系统**：Redo Log（WAL/崩溃恢复）、Undo Log（回滚/MVCC）、Binlog（主从复制/数据恢复）两阶段提交
4. **SQL 优化**：EXPLAIN 执行计划（type/key/rows/Extra）、慢查询定位、回表、文件排序、临时表
5. **主从复制与读写分离**：异步/半同步/组复制、主从延迟处理、分库分表（ShardingSphere）
6. **InnoDB 内存结构**：Buffer Pool（LRU）、Change Buffer、Adaptive Hash Index、Double Write

## 二、原理说明

### B+Tree 为什么比 B-Tree 更适合 MySQL 索引？

- B+Tree 非叶子节点只存 key 不存数据，**单个节点能容纳更多 key**，树高更低（3 层即可存千万级数据），磁盘 IO 更少。
- 叶子节点通过双向链表串联，**范围查询只需定位首尾后顺序扫描**，无需回溯。
- 所有查询最终落到叶子节点，**查询路径长度稳定**，性能可预测。

### MVCC 实现原理

```
ReadView = { m_ids(活跃事务列表), min_trx_id, max_trx_id, creator_trx_id }

可见性判断：
  trx_id == creator_trx_id        → 可见（自己的修改）
  trx_id < min_trx_id             → 可见（事务已提交）
  trx_id >= max_trx_id            → 不可见（在 ReadView 之后才启动）
  min_trx_id <= trx_id < max_trx_id →
      trx_id 在 m_ids 中 → 不可见（事务仍活跃）
      trx_id 不在 m_ids 中 → 可见（事务已提交）
```

- **RC 级别**：每次 SELECT 都生成新 ReadView，所以能看到其他已提交事务的最新修改。
- **RR 级别**：只在事务第一次 SELECT 时生成 ReadView，后续复用，保证可重复读。

### Redo Log / Undo Log / Binlog 协作流程（UPDATE 场景）

```
1. 从 Buffer Pool 读取数据页（不在则从磁盘加载）
2. 记录 Undo Log（用于回滚）
3. 修改 Buffer Pool 中的数据页（脏页）
4. 写 Redo Log Buffer → 按策略刷盘（innodb_flush_log_at_trx_commit）
5. 事务提交：Redo Log 刷盘（prepare）→ 写 Binlog → Redo Log commit（两阶段提交）
6. 后续由后台线程将脏页刷回磁盘（Checkpoint 机制）
```

## 三、面试高频题

**Q1：一条 SQL 执行很慢，你怎么排查？**
→ 先看 EXPLAIN（type 至少 range，key 有命中，rows 估算不大）→ 检查是否走了索引、是否回表、是否有 Using filesort/temporary → 看慢查询日志 → 检查表大小、是否需要分表 → 观察锁等待（show engine innodb status）。

**Q2：为什么 MySQL 用 B+Tree 不用红黑树或 Hash？**
→ 红黑树树高随数据线性增长，磁盘 IO 多；Hash 只支持等值查询，不支持范围/排序/最左前缀；B+Tree 三层存千万、支持范围、磁盘 IO 少。

**Q3：什么情况下索引会失效？**
→ 对索引列做函数/运算/隐式类型转换；LIKE 以 % 开头；违反最左前缀（联合索引跳过前列）；OR 连接非索引列；MySQL 优化器判断全表扫描更快；IS NOT NULL / != 在某些场景。

**Q4：RR 级别下 MySQL 能完全防止幻读吗？**
→ InnoDB 通过 MVCC + Next-Key Lock（间隙锁+行锁）在快照读下通过 MVCC 避免幻读，当前读（SELECT FOR UPDATE/INSERT/UPDATE）通过 Next-Key Lock 锁区间防幻读。特殊场景下仍可能出现（如先快照读后当前读）。

**Q5：Buffer Pool 淘汰策略？**
→ 改进 LRU：young 区（热数据，5/8）+ old 区（冷数据，3/8）。新数据页先进入 old 区头部，首次访问后等待 `innodb_old_blocks_time`（默认1s）才移到 young 区，防止全表扫描污染 Buffer Pool。

# M2 · 消息队列（MQ）

## 一、核心知识点

1. **消息模型**：点对点（Queue）vs 发布订阅（Topic）、推拉模型、消费组（Consumer Group）
2. **消息可靠性**：生产端确认（confirm/ack）、持久化、消费端手动 ACK、消息重试与死信队列（DLQ）
3. **消息顺序性**：局部有序（按 key 分区）vs 全局有序（单分区，性能差）
4. **消息积压处理**：扩容消费者、临时 topic 转发、监控告警
5. **消息重复与幂等**：At-least-once 带来重复消费、业务幂等（唯一键/去重表/状态机）
6. **高可用架构**：Kafka ISR/Controller 选举、RabbitMQ 镜像队列、RocketMQ Broker 主从/Dledger
7. **应用场景**：异步解耦、流量削峰、最终一致性、日志收集、事件驱动

## 二、原理说明

### Kafka 架构要点

```
Producer → [Topic (多个 Partition)] → Consumer Group
               ↓            ↓
          Leader Replica   Follower Replicas (ISR 列表)
               ↓
          ZK/KRaft 管理 Controller、分区 Leader 选举
```

- **Partition 是并行度单位**：一个 Partition 在同一 Consumer Group 中只能被一个 Consumer 消费，但一个 Consumer 可消费多个 Partition。
- **ISR（In-Sync Replicas）**：与 Leader 同步延迟在 `replica.lag.time.max.ms` 内的 Follower。Leader 故障时从 ISR 中选举新 Leader（通过 Controller 协调），保证 **不丢消息**（acks=all + min.insync.replicas≥2）。
- **消费语义**：
  - at-most-once：offset 先提交再处理（可能丢）
  - at-least-once：处理完再提交 offset（可能重复，默认方案，需幂等）
  - exactly-once：幂等生产者 + 事务（Kafka 0.11+，跨分区需事务）

### 如何保证消息不丢失？

| 环节 | 可能丢失 | 解决方案 |
|------|----------|----------|
| 生产端 | 网络抖动发送失败 | confirm 机制 + 本地消息表/重试 |
| Broker | 未持久化就宕机 | 持久化刷盘 + 多副本（acks=all） |
| 消费端 | 处理前就 ACK | 手动 ACK，处理完再提交 offset |

## 三、面试高频题

**Q1：如何保证消息的顺序消费？**
→ Kafka 层面：发送时指定相同 key（如订单ID），保证同 key 消息落入同一 Partition；单 Partition 单 Consumer 消费。业务层面：如果需要全局有序只能用单 Partition（牺牲性能）。注意：Consumer 内部如果多线程异步处理会破坏顺序，需在内存按 key 分发到不同队列串行消费。

**Q2：消息积压了几百万条，怎么处理？**
→ 1）先定位原因（消费变慢/消费失败循环重试/上游突发流量）；2）临时扩容消费者数量（不超过 Partition 数，否则多余消费者空闲）；3）如果 Partition 不够，临时创建新 topic 并扩容 Partition，把积压消息转存到新 topic 用更多消费者并行消费；4）处理完后恢复原架构。

**Q3：如何设计一个幂等的消费逻辑？**
→ 方案一：数据库唯一键约束（最简单，插入重复直接报错）；方案二：Redis setnx / 状态机（订单状态只能单向流转）；方案三：引入去重表（消息 ID + 唯一约束）。关键是在**业务层面**保证幂等，MQ 层只能做到 at-least-once。

**Q4：Kafka 和 RocketMQ 怎么选？**
→ Kafka：高吞吐、生态丰富、适合日志/大数据/流处理；RocketMQ：金融级可靠性、事务消息、延时消息、重试/死信队列更完善、适合业务系统/电商。RabbitMQ：路由灵活、适合中小规模复杂路由场景。

**Q5：什么是事务消息？适用什么场景？**
→ RocketMQ 半消息机制：发送半消息（消费者不可见）→ 执行本地事务 → 根据结果提交或回滚半消息。如果回查超时则回调检查本地事务状态。适用下单+扣库存+派单等跨服务最终一致性场景。

# M3 · Redis 缓存架构

## 一、核心知识点

1. **数据结构与底层编码**：String/Hash/List/Set/ZSet，以及 SDS、ziplist/listpack、skiplist、intset、hashtable 编码转换
2. **持久化**：RDB（快照）、AOF（日志追加）、混合持久化（4.0+）
3. **内存淘汰**：8 种策略（noeviction/allkeys-lru/allkeys-lfu/volatile-ttl 等）、近似 LRU/LFU 实现
4. **缓存三大问题**：缓存穿透、缓存击穿、缓存雪崩的成因与解法
5. **高可用**：主从复制、Sentinel（哨兵）、Cluster（16384 槽位/Gossip 协议）
6. **分布式锁**：SET NX EX + Lua 原子释放、Redlock 争议、Redisson 看门狗
7. **应用场景**：缓存、计数器、分布式锁、排行榜（ZSet）、消息队列（PubSub/Stream）、位图/HyperLogLog/Geo

## 二、原理说明

### 缓存三大问题对比

| 问题 | 现象 | 根因 | 解决方案 |
|------|------|------|----------|
| **穿透** | 查询不存在的数据，请求全量打到 DB | 缓存和 DB 都没有 | ① 布隆过滤器拦截不存在的 key；② 缓存空值（设短 TTL）；③ 参数校验 |
| **击穿** | 热点 key 过期瞬间大量请求打到 DB | 热点 key 失效 + 高并发 | ① 互斥锁（SETNX），只让一个线程回源；② 热点 key 永不过期（逻辑过期+后台异步刷新） |
| **雪崩** | 大量 key 同时过期 或 Redis 宕机 | TTL 集中 / 单点故障 | ① TTL 加随机偏移；② Redis 高可用（Cluster/Sentinel）；③ 多级缓存（本地缓存+Redis）；④ 熔断降级 |

### Redis 分布式锁的正确姿势

```
加锁（原子）：SET lock_key unique_value NX EX 30
    - NX：不存在才加，保证互斥
    - EX：过期时间，防止死锁
    - unique_value：请求标识，防止误删别人的锁

解锁（Lua 原子操作）：
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end

续期（看门狗）：
Redisson 默认每 10s（TTL/3）检查锁持有状态，自动续期至 30s，
业务宕机则锁自然过期。
```

### Redis Cluster 槽位分配

- 16384 个槽位均匀分布到多台 Master 节点，每个 key 通过 `CRC16(key) % 16384` 定位槽位。
- Gossip 协议在节点间传播槽位/状态信息，去中心化。
- **限制**：不支持跨槽位多 key 操作（如 MGET 跨节点），事务/Lua 脚本中 key 必须在同一节点。Hash Tag（`{order}:123`）可让多个 key 落入同一槽位。

## 三、面试高频题

**Q1：Redis 单线程为什么还这么快？**
→ ① 纯内存操作（微秒级）；② IO 多路复用（epoll）单线程处理大量连接，避免线程切换开销；③ 高效数据结构（SDS/skiplist/ziplist）；④ 6.0 后网络 IO 多线程，但命令执行仍是单线程（线程安全）。注意：Redis 6.0 之前是完全单线程，持久化/删除大 key 等阻塞操作在后台线程（bio）执行。

**Q2：Redis 的过期键删除策略？**
→ **惰性删除**：访问时检查过期才删（CPU 友好，内存不友好）+ **定期删除**：每 100ms 随机抽样一批设置了 TTL 的 key 检查，过期则删除（平衡 CPU/内存）。两者配合，仍可能残留过期 key，所以内存淘汰策略兜底。

**Q3：Redlock 是分布式锁的银弹吗？**
→ Martin Kleppmann 与 antirez 的著名争论。Redlock 基于多独立 Master 节点多数派加锁，但存在时钟漂移/GC 停顿/网络延迟等问题可能导致锁失效。实际工程中：单实例 + 主从延迟容忍场景下 SETNX + 看门狗足够；金融级强一致场景用 ZooKeeper/etcd（基于临时顺序节点+会话机制的锁更可靠）。

**Q4：大 key 问题怎么发现和解决？**
→ 发现：redis-cli --bigkeys / MEMORY USAGE / 监控 RDB 文件。危害：删除阻塞（Redis 4.0+ 可用 UNLINK 异步删除）、迁移阻塞、网络阻塞。解决：拆分（Hash 拆小、List 按时间分片）、压缩（MessagePack/Snappy）、冷热分离（大 value 存对象存储，Redis 只存指针）。

**Q5：如何设计一个高并发的排行榜系统？**
→ 核心用 ZSet：ZADD 更新分数、ZREVRANGE/ZREVRANK 查 TopN 和排名。瓶颈与优化：① 实时写入用 ZSet，查询层加缓存；② 分段排行（日榜/周榜/总榜），定时用 ZUNIONSTORE 聚合；③ 亿级数据分桶 + 定时合并；④ 最终一致性可以异步更新。

# M4 · 分布式系统

## 一、核心知识点

1. **CAP 与 BASE**：CAP 定理（一致性/可用性/分区容忍性三选二）、BASE 理论（基本可用/软状态/最终一致性）
2. **一致性协议**：Paxos（Basic Paxos / Multi-Paxos）、Raft（Leader Election + Log Replication + Safety）、ZAB（ZooKeeper）
3. **分布式 ID**：Snowflake（64bit 自增/时钟回拨问题）、数据库号段模式（Leaf）、UUID、Redis INCR
4. **服务治理**：服务注册发现（Nacos/Eureka/Consul）、负载均衡（轮询/加权/一致性哈希）、熔断降级（Sentinel/Hystrix/Resilience4j）、限流（令牌桶/漏桶/滑动窗口）
5. **API 网关**：路由、鉴权、限流、灰度发布、协议转换
6. **分布式追踪**：TraceID/SpanID 透传、OpenTelemetry、Jaeger/Zipkin
7. **一致性模型**：强一致（线性一致/顺序一致）、弱一致（最终一致/因果一致/读己之写）

## 二、原理说明

### Raft 协议核心流程

```
三种角色：Follower → Candidate → Leader
           ↓ Election Timeout 超时
        发起选举，获得多数票成为 Leader
           ↓
        Leader 通过心跳维持权威，所有写操作走 Leader

日志复制：
1. Client → Leader 发送写请求
2. Leader 追加本地日志（uncommitted）
3. Leader 通过 AppendEntries RPC 并行复制到 Follower
4. 多数 Follower 确认后 Leader commit，返回 Client
5. 后续心跳通知 Follower 也 commit

安全性保证：
- 选举限制：只有拥有最新/最长日志的节点才能当 Leader
- Leader-Completeness：已提交的日志一定在未来 Leader 中
```

### 一致性哈希原理

```
普通哈希：hash(key) % N，N 变化导致大量 key 重新映射 → 缓存雪崩
一致性哈希：
  1. 将 0~2^32 映射成环
  2. 对节点（IP/ID）哈希放到环上
  3. 对 key 哈希，顺时针找最近的节点
  4. 新增/删除节点只影响相邻区间的 key

虚拟节点：一个物理节点对应环上多个虚拟点（100-200个），
解决数据倾斜问题，让负载更均匀。
```

### 常见限流算法

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 固定窗口 | 单位时间内计数 | 简单 | 窗口边界突发2倍流量 |
| 滑动窗口 | 细分小窗口滑动统计 | 平滑 | 实现稍复杂，内存占用 |
| 漏桶 | 固定速率流出 | 平滑流量 | 应对突发差 |
| 令牌桶 | 固定速率放令牌，取到即通过 | 允许一定突发 | 实现稍复杂（Guava RateLimiter） |

## 三、面试高频题

**Q1：CAP 三者为什么不能同时满足？举实际例子。**
→ 发生网络分区（P）时，必须在 C 和 A 之间选：如果要 C（数据一致），就要拒绝部分请求（牺牲 A）；如果要 A（始终响应），不同节点可能返回不同数据（牺牲 C）。CP 例子：ZooKeeper/etcd/HBase（分区时少数派不可用）；AP 例子：Eureka/Cassandra/DNS（返回可能旧数据但可用）。注意：没有 P 的情况就是单机，CA 在分布式下不存在。

**Q2：Raft 和 Paxos 有什么区别？为什么 Raft 更易理解？**
→ Paxos 以"证明正确性"为中心，Basic Paxos 只对单个值达成共识，Multi-Paxos 工程实现复杂；Raft 刻意为"可理解性"设计，把问题拆为三个子问题（Leader 选举、日志复制、安全性），强化 Leader 角色，规定日志只能从 Leader 流向 Follower。etcd/TiKV/Consul 都用 Raft。

**Q3：微服务之间怎么保证调用链追踪？**
→ TraceID（全局唯一请求ID）+ SpanID（当前调用ID）+ ParentSpanID（上游调用ID）通过 HTTP Header / gRPC Metadata / MQ 消息头透传。OpenTelemetry 统一了 Trace/Metrics/Logs 的标准 API，数据上报到 Jaeger/Zipkin/SkyWalking 展示。关键在于**跨线程/跨进程/跨MQ**时都要做好 Context 传递。

**Q4：Snowflake ID 有什么问题？怎么解决？**
→ 64位 = 1位符号位 + 41位时间戳(ms) + 10位机器ID + 12位序列号。问题：① 时钟回拨会产生重复 ID（百度 UidGenerator/Magicnumber 用 NTP 检测+等待或抛异常；Leaf 用 ZK 校验时钟）；② 机器 ID 配置冲突（用 ZK/数据库自动分配 WorkID）；③ 41位时间戳只到 2039 年（可扩展位数或自定义纪元）；④ 单节点 4096/ms ≈ 400w QPS，通常够用。

**Q5：什么是服务雪崩？怎么预防？**
→ 服务A → B → C，C 故障导致 B 线程池被阻塞等待 C 响应，B 资源耗尽进而影响 A，故障沿调用链蔓延。预防：① 熔断器（Sentinel/Hystrix）：失败率达阈值则快速失败，开 fallback 降级逻辑；半开状态探测恢复；② 超时+重试（设合理超时，避免重试风暴）；③ 舱壁模式（线程池/信号量隔离）；④ 限流保护；⑤ 异步解耦。

# M5 · 分布式事务

## 一、核心知识点

1. **本地事务 ACID**：InnoDB 事务隔离级别（RU/RC/RR/SERIALIZABLE）及问题（脏读/不可重复读/幻读）
2. **分布式事务理论**：CAP/BASE、两阶段提交（2PC）、三阶段提交（3PC）、TCC、Saga、本地消息表、事务消息
3. **Seata 框架**：AT 模式（自动生成反向 SQL/UNDO_LOG）、TCC 模式、Saga 模式、XA 模式
4. **TCC 设计要点**：Try/Confirm/Cancel 三阶段、空回滚/幂等/悬挂问题
5. **最终一致性方案选型**：强一致（XA/2PC，性能差）、TCC（性能好侵入性大）、Saga（长事务/补偿）、消息最终一致（最常用）
6. **幂等设计**：去重表、唯一键、状态机、token 机制

## 二、原理说明

### 2PC（两阶段提交）流程

```
协调者（Coordinator）           参与者（Participants）
         │                            │
  ┌──────┴─────── 阶段1：Prepare ────┴──────┐
  │ 发送 Prepare                 写 Redo/Undo |
  │                              锁住资源     │
  │                              返回 Yes/No  │
  └──────────────────────────────────────────┘
                    │
         全部 Yes? ─┴─ 任一 No
         ↓              ↓
  ┌──────┴───── 阶段2：Commit/Rollback ──────┐
  │ Commit/Rollback              提交/回滚    │
  │ 通知所有参与者               释放锁       │
  └──────────────────────────────────────────┘

问题：① 同步阻塞（所有参与者锁资源等待协调者指令）；
     ② 单点故障（协调者挂了参与者永久阻塞）；
     ③ 数据不一致（Commit 阶段网络异常部分参与者没收到 Commit）；
     ④ XA 协议就是 2PC 的工业实现（MySQL XA）。
```

### TCC 三大坑与解决方案

```
业务场景：A 扣钱 → B 加钱

Try（预留资源）：A 扣冻结金额；B 增加可入账金额（不真正加钱）
Confirm（确认）：A 冻结金额正式扣除；B 金额正式入账
Cancel（取消）：A 冻结金额返还；B 可入账金额撤销

三大坑：
1. 空回滚：Try 没收到（网络丢包）就收到 Cancel → Cancel 需要识别 Try 是否执行过
2. 幂等：Confirm/Cancel 可能被重复调用 → 事务状态表记录执行状态
3. 悬挂：Cancel 先于 Try 到达（网络拥堵）→ Try 执行时事务已被 Cancel，需识别并拒绝

解决方案：
- 独立事务控制表（xid + branch_id + 状态：INIT/TRIED/CONFIRMED/CANCELED）
- Try 执行前检查状态，已 Cancel 则拒绝（防悬挂）
- Cancel 检查状态，无 TRY 记录则记录空回滚标记
- 所有操作幂等（基于状态机 CAS 更新）
```

### 本地消息表（异步确保）

```
1. 业务操作 + 写消息（到本地消息表）在同一个本地事务中
2. 后台线程轮询消息表，把"待发送"消息投递到 MQ
3. MQ 回调或消费者 ACK 后更新消息状态为"已发送"
4. 消费端幂等处理消息，失败则 MQ 重试 + 死信告警

优点：实现简单、可靠性高、适合最终一致性
缺点：消息表与业务耦合、每库需绑定、高并发下消息表有压力
```

## 三、面试高频题

**Q1：2PC、3PC、TCC、Saga 有什么区别？适用场景？**
→ 2PC：强一致但同步阻塞，适合传统单体跨库（XA）；3PC 引入超时机制 + CanCommit 阶段减少阻塞但仍有不一致可能；TCC：业务层两阶段，性能好但侵入性强（要写3个接口），适合核心资金类短流程；Saga：长事务拆成多个本地事务+反向补偿，适合跨服务长流程（如旅行预订：机票→酒店→租车，失败按反向顺序补偿），但隔离性差（需在业务层加锁/Semantic Lock）。

**Q2：Seata AT 模式怎么实现无侵入分布式事务？**
→ ① 拦截业务 SQL，执行前查询前镜像（before image）；② 执行业务 SQL；③ 查询后镜像（after image）；④ 把前后镜像+业务 SQL 写入 UNDO_LOG 表（和业务 SQL 在同一本地事务）；⑤ 获取全局锁，提交本地事务；⑥ 全局事务提交则异步删除 UNDO_LOG；⑦ 全局回滚则根据 UNDO_LOG 生成反向 SQL 补偿。注意：AT 模式默认是读未提交隔离级别，需要 SELECT FOR UPDATE 获取全局锁。

**Q3：TCC 的空回滚、悬挂、幂等怎么处理？**
→ 用事务控制表（xid+branch_id 唯一键）记录事务分支状态：① 幂等：执行 Confirm/Cancel 时检查状态，已完成则直接返回；② 空回滚：Cancel 时如果没有 Try 记录，插入空回滚标记后返回；③ 悬挂：Try 执行时检查是否已有 Cancel/空回滚记录，有则拒绝执行。

**Q4：为什么互联网公司都用最终一致性而不是强一致？**
→ 性能：2PC/XA 锁资源时间长，吞吐低、延迟高，无法支撑互联网高并发；可用性：网络分区下强一致系统（CP）会拒绝服务，不符合互联网高可用要求；业务可接受：大部分业务（下单/减库存/发优惠券）允许短暂不一致，只要最终一致即可。只有资金、账户类核心场景才追求强一致或 TCC。

**Q5：你们项目里分布式事务怎么落地的？**
→ 按场景选型：① 跨库但在同一服务 → 本地事务或 Seata AT；② 跨服务核心链路（如支付）→ TCC 或 RocketMQ 事务消息；③ 跨服务非核心链路（如下单成功发通知/加积分）→ 本地消息表 + MQ 最终一致；④ 长流程编排 → Saga 状态机（Seata Saga / 自研状态机）。关键是设计幂等+重试+告警兜底。

## 二、原理说明

### Docker 镜像为什么这么小？分层原理

```
镜像 = 多个只读层（Layer）堆叠 + 1个读写层（容器运行时）
每层是文件系统差异（只存增删改）

Dockerfile 指令与层对应：
  FROM ubuntu:20.04         # 基础层
  RUN apt-get update        # 层1：装包
  COPY . /app               # 层2：复制代码
  RUN pip install -r req    # 层3：装依赖
  CMD ["python", "app.py"]  # 元数据层（不产生新层）

层缓存：某层未变则复用缓存，所以 Dockerfile 要把变动少的层放前面。
多阶段构建：
  阶段1（builder）：装编译工具链、打包产物
  阶段2（runtime）：只 COPY 最终产物到精简基础镜像，抛弃编译环境
```

### 容器与虚拟机区别

| 维度 | 容器 | 虚拟机 |
|------|------|--------|
| 隔离级别 | 进程级（共享宿主内核） | OS 级（独立内核） |
| 启动速度 | 毫秒~秒级 | 秒~分钟级 |
| 镜像大小 | MB 级 | GB 级 |
| 性能开销 | 接近原生（<5%） | Hypervisor 开销（10-20%） |
| 安全性 | 弱（共享内核，容器逃逸风险） | 强（硬件辅助虚拟化） |
| 资源密度 | 单机上千容器 | 单机几十 VM |

## 三、面试高频题

**Q1：Docker 的 Namespace 和 Cgroups 分别做什么？**
→ Namespace 做"隔离"：让容器看起来有自己独立的 PID、网络、文件系统、主机名等资源视图（6种 Namespace）。Cgroups 做"限制"：限制一组进程最多使用多少 CPU/内存/磁盘IO/PID数量，并支持优先级和统计监控。两者配合实现容器的"轻量虚拟化"。

**Q2：容器内 PID 为 1 的进程挂了为什么整个容器退出？**
→ PID Namespace 下，PID=1 的进程是容器的 init 进程。容器本质是宿主上的一组被隔离的进程，Docker 通过监控 PID 1 的生命周期判断容器是否存活。PID 1 退出→ Namespace 内没有其他守护进程→ Docker 将容器视为退出。另外 Linux 下 PID 1 有特殊职责（如回收孤儿进程），kill 信号默认不捕获，所以业务进程要正确处理 SIGTERM。

**Q3：怎么优化 Docker 镜像大小？**
→ ① 选小基础镜像（alpine 5MB、distroless、scratch 空镜像）；② 多阶段构建分离编译和运行环境；③ 合并 RUN 指令减少层数（`&&` 连写）+ 清理缓存（`apt-get clean && rm -rf /var/lib/apt/lists/*`）；④ COPY 而非 ADD（ADD 自动解压不安全）；⑤ .dockerignore 排除无关文件；⑥ 使用 dive 工具分析镜像层内容。

**Q4：K8s 中 Pod 和 Container 是什么关系？为什么要有 Pod？**
→ Pod 是 K8s 最小调度单位，一个 Pod 可以包含 1~N 个容器。为什么有 Pod？① 某些应用天生需要紧耦合（如 web 服务 + 日志采集 sidecar / 服务网格 Envoy 代理），它们需要共享网络（同一 IP/端口空间）和存储（共享 Volume）；② Pod 提供"逻辑主机"抽象，容器内共享 Namespace（IPC/Network/UTS），可通过 localhost 通信；③ Pod 内容器总是一起调度、一起生死。Pause 容器作为 Pod 的"根容器"，持有所有 Namespace 和 Volume，业务容器 join 进去。

**Q5：容器逃逸怎么防范？**
→ ① 不以 root 运行（USER 指令/docker run --user）；② 禁用 privileged 模式（不共享宿主所有设备）；③ 限制 capabilities（只给 NET_BIND_SERVICE 等必要权限）；④ 只读根文件系统（--read-only）+ tmpfs 挂可写目录；⑤ 禁止挂载宿主 Docker Socket（/var/run/docker.sock）；⑥ 使用 Seccomp/AppArmor/SELinux 限制系统调用；⑦ 镜像扫描（Trivy）；⑧ 定期更新 runc/containerd 修补漏洞。

## 二、原理说明

### Self-Attention 计算

```
Attention(Q,K,V) = softmax(Q·K^T / √d_k) · V

Q = X·W_q  (查询)
K = X·W_k  (键)
V = X·W_v  (值)

步骤：
1. 线性投影得到 Q/K/V
2. Q·K^T：计算每个 token 对所有 token 的注意力分数
3. /√d_k：缩放，防止点积过大导致 softmax 梯度消失
4. Mask（可选）：Decoder 中掩码未来位置（causal mask）
5. softmax：归一化为权重
6. ·V：加权求和得到每个 token 的新表示

Multi-Head：h 个独立 Attention 头并行计算，最后 Concat + 线性投影，
让模型同时关注不同子空间的信息。

计算复杂度：O(n²·d)，n 是序列长度，这是长上下文的瓶颈。
```

### RAG 完整流程图

```
用户提问
  ↓
[Query 改写层] ← 可选：HyDE/Multi-Query/Step-Back/Rewrite
  ↓
[检索层]
  ├── 向量检索（Dense Retrieval）：Embedding→向量库 ANN 查询→Top-K
  ├── 关键词检索（BM25/Sparse Retrieval）：倒排索引精确词匹配
  └── Metadata 过滤：时间/来源/权限
  ↓
[Rerank 层] ← Cross-Encoder 精排，提升 precision@k
  ↓
[Context 构建] ← 上下文压缩/摘要/引用标注
  ↓
[Prompt 组装] ← System Prompt + 检索上下文 + 用户问题 + Few-shot
  ↓
[LLM 生成] ← 流式输出，约束回答格式，强制引用来源
  ↓
[后处理] ← 答案校验/Fact Check/Guardrails/敏感词过滤
  ↓
返回答案 + 引用来源
```

### LoRA 原理详解

```
W₀ ∈ R^{d×k} 是预训练权重（冻结）

前向传播时：
  h = W₀·x + (B·A)·x
  B ∈ R^{d×r}, A ∈ R^{k×r}  （注意维度顺序，实际实现略有不同）
  r << min(d,k)，如 r=8，d=4096, k=4096
  → 可训练参数：d·r + r·k ≈ 2·4096·8 = 65536
  → 原参数：4096·4096 ≈ 1677万
  → 压缩比约 256 倍

训练时：
  - A 用小方差高斯初始化，B 初始化为 0 → 训练开始时 BA=0，不影响预训练模型
  - 只更新 A 和 B 的梯度

推理时：
  - 可以将 BA merge 到 W₀ 中（W = W₀ + BA），推理零开销
  - 也可以保持分离，方便快速切换 LoRA adapter（多租户/多风格场景）

为什么有效？
  论文证明预训练大模型是"低秩适配"的——
  下游任务的权重更新本质上在一个低维子空间里，
  低秩矩阵足以表达核心变化。
```

### ReAct Agent 循环

```
用户问题 → [LLM 推理]
              ↓
         Thought: 我需要搜索 X 来回答
              ↓
         Action: search("X")     ← 工具调用
              ↓
         [执行工具] → Observation: "搜索结果是..."
              ↓
         [LLM 基于观察继续推理]
              ↓
         Thought: 我现在知道了，最终答案是...
              ↓
         Final Answer: ...       ← 终止条件

LangGraph 实现 = 状态机：
  nodes: {agent, tool_node}
  edges: agent → tool_node (if tool call)
         tool_node → agent
         agent → END (if final answer)
  state = {messages, tools_results, scratchpad, ...}
  checkpoints: 持久化状态，支持中断/恢复/人工审批
```

## 三、面试高频题

**Q1：解释一下 Self-Attention 机制，为什么要除以 √d_k？**
→ Self-Attention 让每个 token 看到所有其他 token 并加权聚合信息。计算 Q·K^T 时，当 d_k 很大时点积方差很大（≈d_k），导致 softmax 后的梯度变得极小（饱和区），除以 √d_k 把方差拉回到 1，使训练稳定。补充：Multi-Head 是为了让不同头关注不同模式（语法/语义/位置关系等），实践中观察到不同头确实学到了不同的注意力模式。

**Q2：RAG 中检索到的文档不相关怎么办？**
→ 多层优化：① Query 层：HyDE（先生成假设答案再用假设答案检索）、Multi-Query（多角度生成多个查询）、Step-Back（先抽象问题再检索）、Query Rewrite；② 检索层：混合检索（向量+BM25）提升召回、Metadata 过滤、RRF 融合；③ 排序层：Cross-Encoder Rerank 精排；④ 生成层：Prompt 中明确告诉模型"如果检索内容不相关就说不知道"，降低幻觉；⑤ 迭代：收集 bad case 持续优化切片/embedding 模型/数据。

**Q3：RAG 怎么解决大模型幻觉？**
→ RAG 通过提供真实上下文来减少幻觉，但不能完全消除。措施：① 强制引用来源（答案标注来自哪个文档哪一段）；② 限定回答范围（Prompt：只基于提供的上下文回答，不要编造）；③ Fact Check：生成答案后用 LLM-as-Judge 校验每个事实是否能在上下文中找到依据；④ 检索增强：多轮对话改写、补充检索；⑤ 知识图谱校验实体关系；⑥ 对于不确定的内容显式表达不确定性。

**Q4：LoRA、QLoRA、全参微调怎么选？**
→ ① 数据量小（几千~几万条）+ 显存有限：LoRA（单卡 24G 可微调 7B-14B）；② 数据量中等+显存非常有限（消费级显卡如 4090/3090）：QLoRA（4bit 量化+LoRA，单卡 24G 可微调 70B）；③ 数据量大（百万+）+ 计算资源充足（8×A100）+ 追求极致效果：全参微调。实践中：LoRA/QLoRA 配合高质量数据通常能达到全参微调 90-95% 效果，且训练快、可插拔多 adapter，是工业界首选。注意：LoRA 的秩 r、alpha、target_modules（q_proj/v_proj 还是全线性层）对效果影响很大。

**Q5：SFT、DPO、RLHF 有什么区别？**
→ SFT（监督微调）：用高质量对话数据直接教模型怎么回答，是微调第一步；RLHF：人工标注偏好→训练奖励模型 RM→用 PPO 让模型按 RM 偏好优化生成（稳定训练难、工程复杂）；DPO（直接偏好优化）：跳过单独训练 RM，直接用偏好数据优化策略，数学等价于 RLHF 下的最优解但实现简单、训练稳定。2024-2025 年主流做法：SFT + DPO 两阶段即可对齐大部分场景，RLHF 主要用于 OpenAI 等拥有大规模偏好数据的顶级团队。

**Q6：Agent 与普通 ChatBot 的本质区别？**
→ ChatBot 是单轮/多轮对话，模型只做"文本输入→文本输出"；Agent 核心是**自主行动能力**：① 能使用工具（搜索/API/代码执行/数据库查询）获取外部信息；② 能规划分解任务（Plan→Subtasks）；③ 能观察结果并调整策略（ReAct 循环/Reflection）；④ 有记忆（长期/短期/工作记忆）；⑤ 能处理多步任务直到完成。类比：ChatBot 是"只会回答的专家"，Agent 是"能干活的助理"。

**Q7：怎么评估一个 RAG 系统效果？**
→ 分层评估：① 检索层：Recall@K（正确文档是否被召回）、MRR/NDCG、检索延迟；② 生成层：Faithfulness（忠实度，回答是否忠于检索内容，核心反幻觉指标）、Answer Relevancy（答案是否回答了问题）、Context Precision/Recall（上下文是否相关且充分）；③ 端到端：任务完成率、用户满意度、人工标注。工具：RAGAS、TruLens 自动化评估；关键是构建业务真实评测集（Golden Dataset），不能只靠通用 benchmark。

**Q8：LangGraph 相比 LangChain Agent 有什么优势？**
→ LangChain 传统 Agent 是黑盒循环（LLM 内部决定走工具还是结束），难以控制流程、持久状态、加人工审批；LangGraph 用显式**状态机图**建模 Agent：① 节点是具体操作（调用LLM/执行工具/路由决策），边是状态转移；② 条件路由（if-else 分支）清晰可控；③ 内置检查点（Checkpointer）自动持久化状态，支持中断/恢复/时间旅行调试/多轮记忆；④ 支持循环边（ReAct），支持并行节点；⑤ 本质是"把 Agent 从 LLM 内部循环变成外部显式图"，工程可控性大幅提升。2025-2026 年 LangGraph 已成为 Agent 开发的事实标准框架。

**Q9：Agent 生产环境落地的主要挑战是什么？**
→ ① 可靠性：LLM 非确定性输出导致工具调用格式错误/幻觉/死循环（强制结构化输出+重试+最大步数限制+Guardrails）；② 延迟：多步工具调用+长上下文导致响应慢（流式输出+并行工具调用+缓存）；③ 成本：Token 费用随步数/上下文长度累积（精简Prompt/语义缓存/模型路由：简单问题用小模型）；④ 可观测性：多步 Agent 出问题难排查（LangSmith/Langfuse 全链路 Trace）；⑤ 安全性：Prompt Injection 诱导 Agent 执行危险操作（工具白名单+输入审查+人工审批关键操作）；⑥ 状态管理：长会话记忆/多轮工具调用的状态一致性（LangGraph Checkpoint+持久化存储）。

**Q10：MCP（Model Context Protocol）为什么重要？**
→ MCP 是 Anthropic 提出的开放协议，标准化了 LLM 与外部工具/数据源/上下文的连接方式（类比 USB-C 统一外设接口）。解决问题：之前每个 Agent 框架/工具都要定制适配，N×M 集成爆炸。MCP 提供：Tools（可执行动作）、Resources（可读数据）、Prompts（模板化提示）三类原语，统一 Client-Server 架构（stdio/SSE 传输）。意义：Agent 工具生态从"各框架封闭"走向"协议互通"，一个 MCP Server 可被所有支持 MCP 的 Client（Claude Desktop/Cursor/Codex/LangChain 等）直接使用。2026 年已成为 Agent 工具接入的事实标准。

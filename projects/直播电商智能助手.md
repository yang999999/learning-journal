# 🤖 直播电商智能助手（内部 OnCall 提效 + C 端智能客服 一体化）

> **一句话**：一套 AI Agent 底层能力平台，面向两类用户——**内部研发/运营**做 OnCall 排障和自助取数，**C 端直播间用户**做智能客服问答。共享模型/工具/RAG/Guardrail/缓存，通过不同入口和权限隔离复用一套底座。
>
> **可信度**：⭐⭐⭐⭐⭐（完全贴合 TikTok 直播电商 C 端流量业务：商卡/购物袋/预览流/秒杀/竞拍/召回/用户行为）
> **技术含金量**：⭐⭐⭐⭐⭐（RAG + Function Calling + MCP + 多 Agent Supervisor-Worker + HITL + 三层防幻觉 + 高并发）

## 📋 一、项目背景（面试 30 秒自我介绍）

> "我做直播电商 C 端流量承接（商卡/购物袋/预览流/秒杀/竞拍/用户行为存储与召回），过程中发现两个痛点：
> 1. **内部**：OnCall 排障时要跨好几个系统查数据（Prometheus/Loki/CMDB/发布系统/用户行为库），研发每次都要拼参数、切页面，新人 OnCall MTTR 很长；运营产品临时取数也要找研发写 SQL，效率低。
> 2. **C 端**：直播间用户高频问价、问秒杀规则、问竞拍结果，人工客服响应慢、回答口径不统一，秒杀瞬间高峰客服根本接不过来。
>
> 我主导做了一套**统一 AI Agent 底座 + 两个场景**：内部 OnCall/取数助手 + C 端智能客服。底层共享模型路由、RAG、工具接入（本地 @tool + MCP）、Guardrail、缓存、可观测；上层按场景切 Prompt、工具白名单、权限。上线后 C 端客服咨询量下降 40%，OnCall MTTR 下降 35%，运营自助取数覆盖 80% 常见查询。"

**关键数字**（合理可调整）：
- C 端峰值 QPS：5000（秒杀瞬间）
- 内部 OnCall 日均使用：研发/运营 200+ 次查询
- 活动规则/内部文档：200+ 篇
- 首 token 延迟：< 500ms（C 端 SSE 流式）
- C 端问题解决率：85%，人工转接率下降 35%
- OnCall 平均排障时间：从 25 分钟 → 16 分钟

## 🛠️ 三、技术栈（Go + Python 微服务）

| 层 | 选型 | 理由 |
|---|---|---|
| 接入/网关 | Go（Gin + gRPC）| 公司微服务全是 Go，goroutine 扛高并发，对接现有基建顺 |
| Agent 编排 | Python + LangGraph | AI 生态成熟，Supervisor-Worker/HITL 状态机原生支持 |
| 通信 | gRPC（Protobuf）| Go/Python 跨语言强类型，独立部署扩容 |
| Embedding | BGE-M3（本地部署）| 中文好，数据不出域 |
| Rerank | bge-reranker-v2-m3 | 精排提 precision |
| 向量库 | Milvus | 文档量万级，支持 metadata 过滤 |
| 工具协议 | 本地 @tool + MCP 双模式 | 核心链路本地包装快；内部基建（监控/日志）走 MCP 让各团队自己维护 |
| LLM | 云端 DeepSeek/Qwen + 本地 vLLM 兜底 | 成本/效果/可用性平衡 |
| 缓存 | Redis 精确缓存 + 语义缓存 | 降本提速 |
| 可观测 | Langfuse + Prometheus | 全链路 Trace |
| 部署 | K8s + HPA | 秒杀流量波动大，自动扩缩容 |

## 🧩 四、核心技术亮点（面试深挖点）

### 亮点 1：一套底座两个场景，差异只在"权限/工具白名单/Prompt"

面试必讲，体现架构能力：
- **共享**：模型路由、RAG 引擎、Guardrail、缓存、工具接入层、可观测、评估
- **差异**：
  - 入口：飞书 Bot/Slack vs 直播间浮窗
  - 鉴权：SSO+RBAC vs 设备/uid
  - 工具白名单：B 端可以查监控/日志/发布；C 端只能查商卡/规则/个人订单
  - 数据：C 端所有用户数据必须经过**脱敏工具**才能进 Prompt（uid hash、不返回别人订单、价格只能查公开商品）
  - Prompt 人格：B 端是"技术排障助手"风格，C 端是"主播助理"风格
  - Agent 模式：B 端多 Agent 并行 Supervisor-Worker + HITL；C 端单 Agent ReAct（低延迟优先）

### 亮点 2：C 端——高并发 + 三层防幻觉（零价格 P0）

**痛点**：C 端峰值 QPS 5000，价格/库存幻觉 = P0 事故。

**方案**：
- **多级缓存**：
  - ① 精确缓存（同问题+用户+商品，TTL 5-30s，打热点）
  - ② 语义缓存（cos > 0.97 命中，减少 LLM 调用）
  - ③ 本地内存缓存（热门活动规则常驻）
- **模型路由**：简单问价走小模型+模板；复杂规则走大模型
- **异步削峰**：对话/埋点/审计走 MQ 异步写，不阻塞主链路
- **降级**：大模型挂 → 小模型兜底 → 引导转人工
- **三层防幻觉**（价格类零事故核心）：

  ```
  用户提问
    → [第1层 Prompt硬约束（意图层）] System Prompt 写死价格/库存/资格必须调工具
        ↓ 模型偶尔分类错（模糊问法/长上下文注意力稀释）
      [第2层 关键词强制路由（规则层）] 代码正则命中敏感词，无视模型决策，强制走工具
        ↓ 工具返回，模型生成回答
      [第3层 Guardrail输出校验（输出层）] 正则提取回答数字 vs 工具返回范围，不匹配重生成
        ↓
      返回用户（带"实时数据"标识+时间戳）
  ```

  - 第2层关键词维护：配置中心 Apollo/Nacos 热更新；来源 = 产品初始枚举 + 线上问法聚类 + bad case 复盘补词
  - 第3层 Guardrail 还防 Prompt 注入、敏感词、System Prompt 泄露

### 亮点 3：C 端——RAG 检索活动规则 + 实时工具调用结合

- 规则类（秒杀/满减/优惠券/竞拍玩法）→ RAG：
  - 切片策略：按规则段落 + 表格单独切（HTML 表格），竞拍规则独立 chunk 避免跨 chunk 丢上下文
  - Embedding：BGE-M3，多路召回（稠密+稀疏=BM25）→ RRF 融合
  - Metadata 过滤：自动注入直播间 ID + 活动 ID + 活动状态（下线活动不召回）
  - Rerank：bge-reranker 精排 Top5
- 实时类（价格/库存/资格/竞拍进度）→ Function Calling：
  - 问题里含这些字段必须调工具，工具返回结构化 JSON → LLM 组织自然语言
- **RRF 为什么用倒数**：①不同路分数量纲不同没法直接加权；②倒数天然重头部轻尾部（第1名1/61≈0.0164，第100名1/160≈0.0063，衰减快）；③不依赖归一化，超参 k 一个常数搞定，工业界简单可靠
- **并行调用**：System Prompt 明确指引"多个不依赖的工具一次并行调"（比如问两个不同商品价格，一次生成两个 tool_call，省一轮 RTT）

### 亮点 4：C 端——个性化回答（用户行为画像注入）

基于你做的用户行为存储+召回系统：
- 请求进来先调行为查询接口拿标签（观看时长/加购/下单/新老客/会员）
- 标签注入 System Prompt
- LLM 生成个性化回答：新客讲基础规则、加购用户提醒秒杀、老客推会员专属
- 转化率提升 X%（填合理数字）

### 亮点 5：B 端 OnCall——Supervisor-Worker 多 Agent 并行排障 + HITL

**痛点**：单 Agent ReAct 串行 查指标→查日志→查变更→查CMDB，延迟是 4 步相加；OnCall 场景要快。

**方案（LangGraph 黑板模式）**：
- 所有 Worker 共享一个 `OnCallState`（黑板），并行写入结果
- Supervisor 根据告警类型路由 Worker 组合（比如"延迟高"→同时拉 Prometheus + Loki + 发布系统 + 相似历史案例）
- 总延迟 = 最慢 Worker，不是相加
- 高危操作（重启/回滚/改配置）→ HITL（Human-In-The-Loop）中断，飞书卡片发审批，人点通过后 Checkpointer 恢复状态机继续执行
- 关键代码骨架（面试能写出来）：

```python
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, END
from langgraph.checkpoint.memory import MemorySaver

class OnCallState(TypedDict):
    alert_id: str
    alert_message: str
    region: str           # 缺失时由 Supervisor 反问用户
    pid: str | None
    time_range: str | None
    activity_id: str | None
    # Worker 结果
    metrics_result: dict | None
    logs_result: dict | None
    changes_result: dict | None
    similar_cases: list | None
    # 输出
    root_cause: str | None
    suggested_actions: list[str]
    pending_approval: bool  # HITL

# Supervisor 反问补参：关键零散信息（region/pid/time_range/activity_id）不全时，
# 不盲目调工具，先一轮追问用户补全，再进入并行排查
```

### 亮点 6：B 端——零散信息补全追问（Slot Filling）

这是你特别提到的需求，面试时非常加分：
- 用户给的告警信息经常是零散的，比如"华北6订单服务挂了"（有 region/pid 没时间范围），或者"昨天秒杀好像有问题"（有时间没 pid/region）
- 实现：Supervisor 节点先做**槽位检查**，必填槽位（region/pid/time_range）缺失 → 直接返回追问，不触发工具调用
- 追问用 LLM 生成，用自然语言问"还需要你提供一下具体的时间范围和活动ID（如果有），方便我查"
- 非必填槽位（activity_id）有就带、没有就默认查全部
- 追问轮数上限 2 轮，避免无限问（超过就用已有信息先查+说明）

### 亮点 7：工具双模式——本地 @tool vs MCP

- **核心高频/性能敏感工具用本地 @tool**（Python 直接包）：
  - 商卡价格/库存/秒杀资格/竞拍状态/C 端脱敏
  - 好处：低延迟、强类型、不走网络协议转换、能做精细缓存
- **跨团队基建用 MCP**：
  - Prometheus 查询、Loki 日志查询、发布系统、CMDB、用户行为召回
  - 好处：**工具由下游团队自己实现和维护**（比如 SRE 团队维护 Prometheus MCP Server），我们 Agent 只负责接入；协议统一（MCP = AI 界的 USB-C），新工具接入不用改 Agent 代码；工具描述/参数/权限由 MCP Server 自己声明
- 对模型来说**看到的都是 tools**，模型不感知区别；Agent 层根据工具注册来源决定是本地调还是 MCP 协议调

> **面试金句**：MCP 不是替代 Function Calling，是"工具提供方"和"工具使用方"之间的标准化协议。我们自己能掌控 SLA 的核心链路用本地 @tool，跨团队的基础能力用 MCP 让对方自维护，解耦。

### 亮点 8：运营自然语言取数（Text-to-SQL/Text-to-API）

- 基于用户行为召回系统现有接口，把 GMV/观看时长/漏斗转化等查询封装成工具
- 运营自然语言问"昨天GMV最高的3个直播间""看过5分钟没下单的用户数"→ LLM 识别意图+填参数→调召回接口→自然语言回答
- 自助取数覆盖 80% 常见查询，研发临时取数需求大幅下降
- 对 C 端数据同样走脱敏通道

### 亮点 9：可观测 + bad case 闭环

- Langfuse 全链路 Trace：每次用户问题→模型决策→工具入参出参→Guardrail 拦截→最终回答 全记录
- 用户 👍👎 反馈 + 人工客服转接记录自动入 bad case 池
- 每周复盘：补关键词、调 Prompt、优化 RAG 切片、补测试集
- 离线评估：300+ 标注 QA 跑 RAGAS（context precision/recall/answer relevancy）+ 业务准确率，每次发版前回归

## 🎤 面试高频追问 Q&A

### Q0：为什么不全 Go 或全 Python？
> Go 扛高并发网关和对接现有 Go 微服务生态；Python 跑 LangGraph/MCP/RAG 迭代快。Protobuf 定义接口，独立部署扩容，是大厂 AI 落地标准架构。

### Q1：MCP 和 Function Calling 本质区别？为什么不都包成本地 func？
> Function Calling 是**模型和工具之间的调用协议**（模型怎么生成 tool_call）；MCP 是**工具提供方和 Agent 应用之间的服务协议**（Agent 怎么发现/调用/权限管理下游工具）。本地 @tool 走函数调用快、可控；跨团队基建用 MCP 让 SRE/数据等下游团队自维护，Agent 零改代码接入新工具。对模型来说都是 tools，Agent 层做路由。

### Q2：C 端用户隐私怎么处理？
> ① C 端有独立的脱敏工具 @tool，所有用户数据（uid/订单/收货信息）必须经过脱敏才会进 Prompt，uid 用 hash，订单只返回本人的；② Prompt 里严禁模型索要身份证/手机号等 PII；③ Guardrail 输出层拦截敏感字段；④ C 端工具白名单隔离，调不到内部 B 端工具。

### Q3：如果 MCP Server 挂了/慢了怎么办？
> ① 工具调用层有超时（C 端 < 2s，B 端 < 10s）；② 超时/失败→降级到默认文案"暂时无法查询"，C 端引导转人工，B 端建议直接查监控页面；③ 核心工具（商卡价格）本地冗余一份 fallback，不依赖 MCP；④ 熔断：某个 MCP 连续失败 N 次自动熔断一段时间。

### Q4：三层防幻觉的关键词怎么来、怎么热更新？
> 来源三个：①产品/客服初版枚举（价格类/库存类/资格类/竞拍类）；②线上真实用户问法聚类（每周跑一次 query 聚类，发现新模式补词）；③bad case 复盘补词（每一次幻觉事故复盘必补词）。存在配置中心 Apollo/Nacos，改了不用重启服务实时生效。

### Q5：怎么判断大模型"15 次分类对了、5 次分类错了"？
> ① Guardrail 层拦截到必须调工具的问题但模型没调→算分类错，埋点计数；②用户点踩+人工转接的 case 人工抽检标注；③离线测试集跑分类准确率；④线上用关键词兜底触发率做信号（如果关键词兜底触发率突然升高，说明模型分类在退化）。

### Q6：RAG 切片具体怎么切、参数？
> 文档进来后：①Unstructured 解析（支持 Word/Markdown/HTML 表格）；②按语义切片（相近句子合并），chunk_size=512 tokens，overlap=64 tokens；③表格单独处理转 HTML；④每 chunk 带 metadata（直播间ID/活动ID/状态/版本/生效时间）；⑤BGE-M3 编码成 1024 维向量；⑥写入 Milvus 集合（HNSW 索引，metric_type=IP，partition_key=activity_id 方便下线活动快速清理）。查询时：①用户问题同样 BGE-M3 向量化；②带 metadata filter（活动状态=进行中 + 直播间ID）；③Milvus search（top_k=20，nprobe=16）；④bge-reranker 精排取 Top5。

### Q7：B 端多 Agent 为什么不用 Swarm？
> Swarm 是平等对话模式，适合多专家平级讨论；OnCall 场景需要一个 Supervisor 集中做权限管控、路由、HITL 审批决策，Supervisor-Worker 更简单可控。

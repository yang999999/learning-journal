## 🧩 四、核心技术亮点（面试重点讲，每个都能挖10分钟）

### 亮点 1：多策略 Chunking + 父子文档检索

**问题**：固定大小切片切断语义，切片太小没上下文，太大引入噪声。

**方案**：
- Markdown/文档按标题层级递归切片（RecursiveCharacterTextSplitter）
- 代码按函数/类切（Tree-sitter AST解析）
- 表格转 HTML 保留行列结构单独处理
- 父子文档策略：小块（256 token）用于检索（精度高），命中后返回父块（1024 token）给LLM（上下文完整）

**量化效果**：Recall@5 从 74% 提升到 88%。

### 亮点 2：混合检索 + Rerank 多路融合

**方案**：
- 向量检索（BGE-M3，语义匹配）+ BM25关键词检索（Elasticsearch/PG-BM25，精确词匹配）
- RRF（Reciprocal Rank Fusion）融合两路排名，score=1/(k+rank)，k=60
- 多路召回 Top20 → Rerank（bge-reranker）精排 → Top5 给LLM
- 按文档权限/分类/时间 Metadata 预过滤

**量化效果**：相比纯向量，Faithfulness（忠实度）从 81%→92%。

### 亮点 3：Query 理解与改写

**方案**：
- **多轮对话改写**：把指代消解（"它是什么"→"RAG是什么"）用 LLM 做 Query Rewrite
- **HyDE（假设文档嵌入）**：先让 LLM 生成一个"假设答案"，再用假设答案检索，比直接用问题检索召回率更高
- **Query 路由**：简单FAQ→直接检索FAQ库；复杂技术问题→检索技术文档；闲聊→直接回答

### 亮点 4：幻觉治理与答案可溯源

**方案**：
- Prompt 强约束："只根据提供的上下文回答，不知道就说不知道"
- **强制引用来源**：每段答案后标注[来源:文档名#章节]，前端可跳转原文
- **Faithfulness 自动校验**：生成后用 LLM-as-Judge 校验每句话是否都在上下文里找到依据，低于阈值则让模型重写或回复"无法确定"
- 用户反馈闭环：👍👎 按钮收集bad case，持续优化切片/Embedding

### 亮点 5：工程化深度

- **语义缓存**：相似问题命中Redis缓存（用Embedding余弦相似度>0.95），缓存命中率27%，token成本降低32%
- **模型路由**：简单问题（FAQ类）走Qwen2.5-7B本地部署，成本是GPT-4o-mini的1/10；复杂问题走云端大模型
- **流式响应**：SSE打字机效果，首token延迟<300ms
- **多租户/权限**：文档级权限继承（对接公司SSO），检索时按用户权限过滤，A部门看不到B部门文档
- **可观测**：Langfuse全链路Trace，每个问题的检索片段/生成时间/Tokens/Cost都能复盘

## 🎤 六、面试追问 Q&A（必须能答出来）

### Q1：为什么选 BGE-M3 不是 OpenAI text-embedding？
> 答：BGE-M3 是智源开源的多语言多粒度模型，中文效果接近甚至超过OpenAI；最重要是可以本地部署，企业内部文档数据不出域，符合公司合规要求；而且免费，大规模文档Embedding成本为零。

### Q2：向量库为什么选 Milvus？Pgvector 不行吗？
> 答：快速启动/小数据量用Pgvector足够，和PG一套运维体系。我们上线后数据量涨到XX万切片，需要HNSW索引+混合检索+水平扩展，切换到Milvus。**选型要看数据规模，不盲目追新技术**。

### Q3：父子文档怎么实现？
> 答：存两份：父文档（大chunk，1024 tokens，元数据里有parent_id）和子文档（小chunk，256 tokens，metadata里有parent_id指向父文档）。检索时在子文档库做向量匹配，命中后拿parent_id取父文档原文拼到Prompt里。

### Q4：Reranker 为什么比 Bi-Encoder 准？为什么不直接用 Reranker 检索？
> 答：Bi-Encoder（如BGE）是query和doc分别编码→算余弦，速度快但query和doc没有信息交互，适合粗排；Cross-Encoder把query和doc拼在一起过Transformer，有完整的Cross-Attention，精度高但需要N次推理，太慢。所以流程是：Bi-Encoder快召回Top20→Cross-Encoder精排Top5，兼顾速度和精度。

### Q5：混合检索 BM25 和向量怎么融合？为什么不用加权分数？
> 答：BM25分数和向量余弦分数量纲不同，直接加权不好调。RRF（Reciprocal Rank Fusion）基于排名融合：`score = Σ 1/(k+rank)`，k通常取60，不需要调权重，对不同模型鲁棒性好。

### Q6：用户反馈bad case你怎么处理？
> 答：建立bad case分析流程：①分类（漏检/误检/上下文够了但幻觉/格式问题）；②漏检→优化切片/加HyDE/换Embedding；③误检→加Rerank/MMR多样性；④幻觉→优化Prompt/Faithfulness校验；⑤定期回归测试用RAGAS评估。

### Q7：高并发场景怎么优化？
> 答：①多级缓存（精确缓存+语义缓存+本地缓存）；②vLLM PagedAttention推理加速，吞吐提升15倍；③异步工具调用，并发检索；④流式输出用户感知延迟低；⑤K8s HPA根据QPS自动扩缩副本；⑥Embedding批处理。

### Q8：怎么解决文档更新后索引不一致？
> 答：文档有version字段，更新时异步任务删旧向量+切新向量+写新向量，写过程中保持旧索引可用（双写平滑切换）；监听文档变更事件（CDC/文件系统watch）触发增量更新。
